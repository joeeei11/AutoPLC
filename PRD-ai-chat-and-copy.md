# PRD：AI 需求对话助手 + ST 代码复制按钮

## Problem Statement

用户在使用 PLCLogicGen 生成 PLC 逻辑时，面临两个摩擦点：

1. **需求模糊**：用户往往无法一次性写出完整、精确的控制逻辑描述。生成出来的梯形图和 ST 代码往往与实际需求有偏差，需要反复手动修改描述再重新生成，效率低。
2. **代码无法快速复制**：右侧面板的 ST 代码编辑器（Monaco Editor）没有一键复制按钮，用户需要手动全选再复制，操作繁琐。

## Solution

1. **AI 对话 Tab**：在左侧面板新增"对话"Tab，提供一个由资深 PLC 工程师角色驱动的 AI 聊天界面。AI 主动追问，将模糊需求拆解清晰，最终生成一份整理好的需求摘要，用户可一键将摘要填入"生成"Tab 的描述框并触发生成。

2. **ST 代码复制按钮**：在右侧面板的 ST 代码区 Header 添加"复制"按钮，点击后将当前 ST 代码写入剪贴板，并短暂显示"已复制"状态反馈。

## User Stories

1. 作为 PLC 工程师，我希望能在应用内和 AI 对话，以便在填写描述之前把控制逻辑需求梳理清楚。
2. 作为 PLC 工程师，我希望 AI 扮演资深 PLC 工程师角色，以便获得专业的工程建议而不是泛泛的聊天回复。
3. 作为 PLC 工程师，我希望 AI 能主动追问需求细节，以便不用自己知道"应该提供哪些信息"。
4. 作为 PLC 工程师，我希望 AI 在追问到足够清晰时能给出整理好的需求摘要，以便我知道对话可以结束了。
5. 作为 PLC 工程师，我希望点击"填入描述"按钮后摘要自动填入生成 Tab 的描述框，以便无需手动复制粘贴。
6. 作为 PLC 工程师，我希望点击"填入描述"后页面自动切换到"生成"Tab，以便流程顺畅不需要额外操作。
7. 作为 PLC 工程师，我希望 AI 知道我当前填写的描述内容，以便对话可以基于已有信息继续推进而不是从零开始。
8. 作为 PLC 工程师，我希望 AI 知道当前已生成的 ST 代码，以便我可以问"这段代码里的定时器逻辑对吗"之类的问题。
9. 作为 PLC 工程师，我希望 AI 回复以流式方式逐字输出，以便不需要等待整个回复完成才能看到内容。
10. 作为 PLC 工程师，我希望对话使用和生成 Tab 相同的语言模型（由下拉框控制），以便统一管理模型选择。
11. 作为 PLC 工程师，我希望对话历史在切换 Tab 时保留，以便切回"生成"Tab 再切回来不会丢失上下文。
12. 作为 PLC 工程师，我希望能清空对话历史重新开始，以便开始一个全新的需求讨论。
13. 作为 PLC 工程师，我希望在 AI 回复末尾看到"当前需求摘要"区块，以便随时了解 AI 对需求的最新理解。
14. 作为 PLC 工程师，我希望只有当 AI 认为需求已足够清晰时才显示摘要和"填入描述"按钮，以便避免用不完整的摘要覆盖已有描述。
15. 作为 PLC 工程师，我希望在 AI 回复流式输出期间输入框处于禁用状态，以便不会发出并发请求。
16. 作为 PLC 工程师，我希望网络错误时对话界面显示明确的错误提示，以便知道是否需要重试。
17. 作为 PLC 工程师，我希望能在 ST 代码区点击一个复制按钮，以便快速将代码复制到剪贴板而不需要手动全选。
18. 作为 PLC 工程师，我希望复制成功后按钮短暂显示"已复制"状态，以便得到操作反馈。
19. 作为 PLC 工程师，我希望没有 ST 代码时复制按钮处于禁用状态，以便避免复制空内容的困惑。

## Implementation Decisions

### 模块划分

**后端新增：`/api/chat` 流式端点**
- 接收：消息历史列表、当前 `description`（生成 Tab 的描述框内容）、当前 `st_code`（已生成的 ST 代码，可为空）、`llm` 选择
- 响应：Server-Sent Events（SSE）流，每个事件携带一个文本增量
- 系统 Prompt 固定为资深 PLC 工程师角色，要求 AI 主动追问需求，在需求清晰时在回复末尾输出结构化的需求摘要块
- 使用 FastAPI `StreamingResponse` + `openai` / `anthropic` SDK 的流式接口实现；不使用 pydantic-ai（pydantic-ai 目前不支持流式文本输出）
- 消息历史由前端维护，每次请求完整发送给后端，后端无状态

**摘要提取约定**
AI 被要求在需求清晰时，在回复末尾输出如下格式的块：

```
---SUMMARY---
<整理好的需求描述，可直接用于生成>
---END---
```

前端解析此块，将摘要内容单独渲染为可点击的"填入描述"卡片，从原始回复文本中去除该块后正常显示其余内容。

**前端修改：左侧面板 Tab 化**
- `LeftPanel` 拆分为两个子组件：`GenerateTab`（现有控件）和 `ChatTab`（新对话界面）
- Tab 状态由 `LeftPanel` 管理（`"generate" | "chat"`）
- 切换 Tab 时两个子组件均保持挂载（不卸载），避免丢失状态

**前端新增：`ChatTab` 组件**
- 对话消息列表（用户消息 / AI 消息交替显示，流式 AI 消息显示打字中光标）
- 消息输入框 + 发送按钮（流式输出期间禁用）
- 清空历史按钮
- AI 消息末尾若解析到摘要块，渲染"需求摘要"卡片，含"填入描述"按钮
- 点击"填入描述"：调用父组件传入的回调 `onFillDescription(summary: string)`，`App` 中该回调更新 `description` 状态并切换左侧 Tab 到 `"generate"`

**前端修改：`RightPanel` 复制按钮**
- 在 Header 区域的"重新渲染"按钮旁加"复制"按钮
- 使用 `navigator.clipboard.writeText(stCode)` 实现
- 按钮状态：`stCode` 为空时 disabled；复制成功后 2 秒内显示"已复制"，之后恢复原文本

**前端修改：`api.ts`**
- 新增 `streamChat(req: ChatRequest, onChunk: (text: string) => void): Promise<void>`，封装 SSE 读取逻辑
- `ChatRequest` 类型：`{ messages: ChatMessage[], description: string, st_code: string, llm: LLM }`
- `ChatMessage` 类型：`{ role: "user" | "assistant", content: string }`

**后端 CORS**
- 将 `http://localhost:5174` 加入 `CORS_ORIGINS` 环境变量默认值（前端实际端口）

### API 契约

`POST /api/chat`

请求体：
```json
{
  "messages": [{"role": "user", "content": "..."}, ...],
  "description": "当前生成描述（可为空字符串）",
  "st_code": "当前 ST 代码（可为空字符串）",
  "llm": "openai" | "claude"
}
```

响应：`Content-Type: text/event-stream`，每条 SSE 事件格式：
```
data: {"delta": "文本增量"}\n\n
```
流结束时发送：
```
data: [DONE]\n\n
```

## Testing Decisions

好的测试只验证外部可观测行为，不关心内部实现细节（如调用了哪个 SDK 方法、内部状态的中间值）。

**后端测试：`/api/chat` 端点**
- 测试正常情况：发送消息列表，验证响应为 SSE 流，能读到文本增量事件和 `[DONE]` 终止事件
- 测试空消息列表：验证返回 422
- 测试 `llm` 字段非法值：验证返回 422
- 参考现有测试：`tests/test_api.py` 中已有对 `/api/generate`、`/api/validate` 的 `httpx` 同步测试风格

**前端测试**
- 本项目目前无前端测试，本 PRD 不新增前端测试（留作后续）

**不测试的内容**
- AI 回复的质量和内容（LLM 输出不确定，不适合单元测试）
- 流式解析的每一个中间状态
- 剪贴板写入（依赖浏览器 API，不做单元测试）

## Out of Scope

- 对话历史的持久化（页面刷新后丢失属预期行为）
- 多轮对话的后端会话管理（无状态设计，历史由前端传递）
- 对话 Tab 有独立的模型选择下拉框
- AI 主动修改 ST 代码（只负责需求澄清，不直接操作代码）
- 对话消息的 Markdown 渲染（纯文本显示）
- 移动端响应式布局适配

## Further Notes

- 系统 Prompt 中需明确要求 AI 使用中文回复（与项目整体语言一致）
- 摘要块的分隔符（`---SUMMARY---` / `---END---`）需在系统 Prompt 中精确定义，前端解析依赖此约定
- 流式输出期间摘要块可能被分割在多个 chunk 中，前端需要在整个流结束后才解析摘要块，而不是逐 chunk 尝试解析
- 前端 Vite 代理配置已将 `/api` 转发到 `localhost:8000`，新端点无需额外配置
